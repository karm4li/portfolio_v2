<Section label="Overview">

Machine learning models are only as good as the data they receive. When prediction latency is critical — think fraud detection, recommendation engines, real-time pricing — the data pipeline becomes the product.

This project replaced a batch-oriented legacy system with a streaming architecture capable of serving fresh, feature-rich data to ML models in production.

</Section>

<Section label="The Problem">

The existing system processed events in hourly batches. This meant:
- Features were up to 60 minutes stale at prediction time
- Model performance degraded significantly for time-sensitive predictions
- The data engineering team spent 40% of their time on pipeline incidents

<Callout type="insight">
  A model trained on fresh data but served stale features is a model trained on lies. The training–serving skew was costing us more than any model improvement could recover.
</Callout>

The business impact was measurable — and bad enough that it justified a full infrastructure rebuild.

</Section>

<Section label="Architecture">

The new system is built on three layers:

**Ingestion** — Kafka-based event streaming with schema validation using Avro. Events flow from application services into topic partitions organized by entity type.

**Feature computation** — Apache Flink jobs compute features in real time. Windowed aggregations (last N events, rolling sums, etc.) are computed incrementally as events arrive.

**Feature serving** — Redis-backed feature store with a gRPC API. Models query features by entity ID with p99 latency under 8ms.

</Section>

<Section label="Process">

The hardest part wasn't the technology — it was the migration.

We couldn't afford a cutover. The existing system served production traffic, and a cold migration would have meant hours of degraded model performance. Instead, we ran both systems in parallel for six weeks, continuously comparing feature values and reconciling differences.

This dual-write period was invaluable. It surfaced three significant bugs in our feature computation logic that we would have shipped to production otherwise.

The migration was incremental: model by model, we shifted traffic from the batch system to the streaming system, monitoring metrics at each step.

</Section>
